# Flowable 工作流平台构建路线图

基于 Flowable 打造自己的工作流平台，是一条"站在巨人肩膀上"的渐进式路线。下面把整个过程拆成 5 个阶段，每个阶段给出目标、关键技术点、可直接落地的步骤与工具，以及容易踩的坑。适合 1~3 人小团队起步，也适合后续横向扩展成专职中台。

## 阶段 1：技术预研（1 周）

**目标**：验证 Flowable 能不能跑通最复杂的 1~2 条业务线，并输出《技术可行性报告》。

### 1. 跑通最小闭环

- 官方 docker-compose 把 flowable-rest + flowable-ui 启起来。
- 用 Flowable-Modeler 画一条"会签+驳回+转办"流程 → 导出 bar → 调 REST 部署 → 调 REST 启动实例 → 完成审批。
- 用 Postman 集合把以上 6 个 REST 接口固化下来，作为后续性能基准脚本。

### 2. 性能与扩展性摸底

- 把上面 Postman 脚本用 JMeter 压 10 min，50 并发，看 CPU/DB 连接/乐观锁异常。
- 记录"流程实例数 ≈ 50 万"时历史表（ACT_HI_*）占用的磁盘，评估后续分库分表方案。

### 3. 关键源码走读

- 只读三类源码：CommandExecutorImpl、BpmnDeployer、TaskEntityManagerImpl，目的是知道"事务边界"和"插件入口"。
- 用 IDEA 的 "Diagram → Show Dependency" 把 flowable-spring-boot-starter 的自动配置画成一张图，后续自己封装 starter 会抄它。

### 输出物

- 1 份《Flowable 性能基准报告》（含 TPS、RT、DB 规格）。
- 1 张《核心类图 + 事务边界图》。
- 1 份《风险清单》，至少包括：乐观锁重试、历史表膨胀、REST 无租户隔离。

## 阶段 2：总体架构设计（3 天）

**目标**：决定"嵌入式"还是"独立引擎"，以及"流程服务"与"业务服务"怎么拆。

### 1. 部署模式选型

**推荐"独立引擎 + 多租户"模式**：

- 引擎单独一个 SpringBoot 应用（flowable-engine-svc），只暴露 REST/GraphQL。
- 订单、报销、采购等业务服务通过 OpenFeign 调引擎，做到业务库零耦合。

**优点**：升级引擎不重启业务；DB 独立，方便后续做"历史库"归档。

### 2. 技术栈锁定

- **引擎侧**：SpringBoot 2.7 + Flowable 6.8 + MySQL 8.0（row_format=dynamic）+ Redis（idGenerator 用 RedisSequenceGenerator）。
- **业务侧**：任意语言，只要调 REST 即可。
- **流程设计器**：直接内嵌 Flowable-Modeler 的 Angular 包，前端用 iframe 嵌进来，减少 60% 自研工作量。

### 3. 租户模型

采用 Flowable 自带的 TENANT_ID_ 字段，所有表天然支持。

在业务服务调用引擎时统一在 header 带 X-Tenant-Id，网关层写 1 个 TenantContextFilter 把租户塞进 Flowable 的 CommandContext。

### 4. 表扩展策略

不改动 Flowable 原表，只建 1 张 ext_proc_variable 存业务 JSON，主键与 ACT_RU_EXECUTION 一对一，后续升级引擎零冲突。

## 阶段 3：核心功能二次开发（4~6 周）

**目标**：把"流程图设计 → 节点配置 → 提交/审批/驳回/转办/会签/自动签"全链路做成可配置，提供 1 套内部 npm 组件 + REST SDK。

### 1. 流程图设计器

- 把 flowable-modeler 的 war 解压，把 stencilset_bpmn.json 汉化，把 "UserTask" 新增 3 个自定义属性：
  - candidateType（固定/角色/表单变量/自定义SQL）
  - autoComplete（true=自动签）
  - multiInstance（串行/并行/或签）
- 用 Vue3 包装成 bpmn-designer 组件，发布到私有 npm，1 行代码就能嵌入任意前端页面。

### 2. 节点配置

- 写 1 个 Entity：ProcNodeExtend，字段就是上面 3 个自定义属性 + 通知模板 + 表单权限 JSON。
- 监听 ENTITY_CREATED 事件，把 Modeler 里导出的 BPMN 解析，把自定义属性落库。

### 3. 提交 & 审批

- 业务服务只调 1 个通用接口：POST /runtime/process-instances，传 tenantId、businessKey、startFormData。
- 引擎侧写 1 个 GlobalStartProcessCmd，自动做：
  - 把 businessKey 回写到业务表；
  - 根据 startFormData 计算第一个节点的审批人；
  - 发 RocketMQ 消息，业务侧监听后推待办/邮件/企微。

### 4. 驳回 & 转办

- **驳回**：利用 Flowable 的 BoundaryEvent + rollback 命令，封装成 POST /task/{id}/reject?targetActivityId=xxx。
- **转办**：直接调 taskService.setAssignee，同时写 1 条转办记录到 ext_task_delegate 表，前端可查看"我转出的"和"我收到的"。

### 5. 会签

| 模式 | 配置方式 | completionCondition |
| :--- | :--- | :--- |
| 串行 | multiInstance=sequential | - |
| 并行 | multiInstance=parallel | ${nrOfCompleted/nrOfInstances >= 1} |
| 或签 | multiInstance=parallel | ${nrOfCompleted > 0} |

以上 3 种模式在 Modeler 里做成下拉框，节点保存时自动生成 XML，零手写。

### 6. 自动签

- 在 TaskCreatedEvent 里判断 autoComplete=true，则直接调用 taskService.complete(taskId, autoApproveMap)。
- 为防止死循环，加 1 个 ThreadLocal 深度计数器 >3 就抛异常。

### 7. 统一 SDK

给业务侧提供 flowable-client-spring-boot-starter，内置：

- FeignClient + 熔断；
- TenantContext 自动传递；
- @EnableFlowableClient 1 个注解即可注入 ProcessInstanceService、TaskService。

这样各业务线开发感知不到 Flowable，像调本地 Service 一样。

## 阶段 4：监控与运维（并行进行，1 周）

**目标**：让运维愿意接，能报警，能回滚。

### 1. 历史数据归档

用 MySQL 的 partition（按 proc_inst_id_ 哈希 64 分区）+ 每月交换分区到历史库，查询用 Hive/ClickHouse。

### 2. 慢 SQL 治理

把 flowable 自带的 3 条全表扫描 SQL（ACT_HI_ACTINST 按 proc_inst_id_ 查）改写成索引覆盖，建联合索引 (PROC_INST_ID_, ACT_ID_)。

### 3. 引擎升级策略

- 所有表结构变更用 Flowable 自带的 liquibase changelog，不手写；
- 升级流程：shadow 库 diff → 双跑 24h → 流量 1% 灰度 → 100%。

### 4. 监控看板

Grafana 模板已开源（ID 16367），直接导入即可看"流程实例 TPS"、"待办 Top10 用户"、"Job 死信队列"。

## 阶段 5：落地路线与迭代计划

| 时间 | 阶段 | 目标 |
| :--- | :--- | :--- |
| 第 0 个月 | 预研 + 架构 | 完成阶段 1 预研 + 阶段 2 架构评审 |
| 第 1~2 个月 | 核心功能 | 先接 1 条最简单的"采购申请"流程上线（灰度 10%） |
| 第 3 个月 | 功能完善 | 接入第 2 条"合同审批"流程，把会签、或签、自动签全部跑通 |
| 第 4 个月 | 中台化 | 抽象成"流程中台"，提供 SaaS 化控制台 |
| 第 5 个月以后 | 增值功能 | 低代码表单设计器、流程预测、AI 智能审批等 |

## 常见坑汇总

| 坑点 | 原因 | 解决方案 |
| :--- | :--- | :--- |
| 乐观锁异常 | 并发高时 ACT_RU_TASK 更新会冲突 | 调大 retry 次数 + 业务幂等 |
| 历史表膨胀 | 6 个月后单表破 5000W 行 | 做分区或归档 |
| 租户隔离漏传 | A 租户能查到 B 租户流程 | 网关层统一塞 TENANT_ID_ |
| 自动签死循环 | 会签节点也勾了 autoComplete | 加深度计数器 |
| 升级踩坑 | ACT_ID_ 长度从 64→255 | 升级前务必 diff 表结构 |

## 快速开始（Docker）

```bash
git clone https://gitee.com/flowable/flowable-docker.git
cd flowable-docker
docker-compose -f flowable-rest.yml up -d
```

浏览器打开 `http://localhost:8080/flowable-ui`，账号 admin/test。

```bash
# 部署流程
curl -X POST -u rest-admin:test \
  -F file=@leave.bpmn20.xml \
  http://localhost:8080/flowable-rest/service/repository/deployments
```

## 结语

把上面 5 个阶段按节奏推进，就能在 3 个月内拿到一套"可灰度、可扩展、运维愿意接"的工作流中台。后续想加低代码、想加 AI 审批，都只需要在现有骨架上长出新模块，而不会动到核心引擎。
